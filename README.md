# 明辰 Proof

把“我看过页面了”变成一份**别人能复查、能复现、能判断边界**的网页验证记录。

这是一个面向 Codex 的浏览器验真 Skill。它适合检查页面是否打开、按钮是否反馈、登录后功能能否使用、问题能否复现，以及某条 UI 流程是否满足验收条件。它不会把“页面看起来没问题”写成“业务一定成功”，也不会在没有明确许可时替你提交、发布、付款、删除或改动生产数据。

本仓库中的验证流程、证据格式和示例由明辰原创设计与维护，并以 [MIT License](LICENSE) 开源。

## 它能帮你验证什么

| 你想确认的事 | 会留下的证据 |
| --- | --- |
| 页面能否正常打开 | URL、首屏状态、可见错误、关键元素与截图 |
| 按钮是否真的有反馈 | 操作前后状态、提示/跳转、刷新后的结果 |
| 登录后能否使用功能 | 测试身份、前置条件、最小操作路径与权限提示 |
| 某个问题能否复现 | 复现条件、步骤、预期/实际结果与稳定性 |
| 流程是否可验收 | 逐条断言的通过/失败/无法确认、证据与限制 |

## 第一次使用：按这 4 步来

### 1. 安装到 Codex

在终端运行：

```bash
cd ~/Desktop
git clone https://github.com/zhengjj8888-cmyk/mingchen-proof.git
mkdir -p ~/.codex/skills
cp -R mingchen-proof ~/.codex/skills/
```

也可以下载 ZIP，解压后把 `mingchen-proof` 整个文件夹复制到 `~/.codex/skills/`。

### 2. 重启 Codex

重启后，Codex 才会识别 `$mingchen-proof`。

### 3. 把“看看正常吗”写成可验证的断言

不要只说“帮我看一下页面”。请说明谁、在什么条件下、做什么操作、应该看到什么：

```text
使用 $mingchen-proof 验证以下断言：

未登录用户打开 [URL] 后，点击“导出”按钮会看到登录提示，
且不会直接下载文件。

请记录：测试 URL、时间、未登录状态、最小操作步骤、
操作前后截图或快照、可见结果与限制。
不要登录、不要提交表单、不要触发任何外部写入。
```

### 4. 看结论，也看限制

每个断言都应得到“通过、失败或无法确认”。同时检查证据路径、URL、时间和限制：浏览器会话、缓存、权限、网络、测试数据、第三方服务等。

## 四个可以直接复制的提示词

### 检查一个页面是否正常打开

```text
使用 $mingchen-proof 检查 [URL]。

验证断言：页面能加载，首屏能看到 [关键元素]，且没有可见的错误提示。
请先观察并保存初始快照，再给出结论、最小步骤、证据位置和限制。
如果需要登录或页面不可访问，请标为“无法确认”，不要猜。
```

### 验证按钮，但不提交任何内容

```text
使用 $mingchen-proof 验证 [URL] 上的「[按钮名]」。

断言：点击后会出现 [提示/弹窗/跳转]。
只做一次最小操作，保存点击前后证据；不要点击最终提交、付款、发布、删除或确认按钮。
刷新页面后确认状态是否仍然符合断言。
```

### 复现一个问题

```text
使用 $mingchen-proof 复现这个问题：

[问题描述]
前置条件：[登录身份/测试数据/浏览器/网络]
预期结果：[应该发生什么]
实际结果：[用户看到什么]

请记录最小复现路径和每一步可见结果。
若当前条件下未复现，结论写“当前条件下未复现”，并列出可能的差异；
不要把它写成“问题不存在”。
```

### 验证登录与权限

```text
使用 $mingchen-proof 验证 [测试身份] 在 [URL] 是否能使用 [功能]。

断言：[具体行为与预期权限提示]。
只使用已经登录的测试会话，不要记录密码、cookie、令牌或个人数据。
不要绕过权限、不要尝试提权、不要执行任何会修改生产数据的最终操作。
输出结论、步骤、证据和边界。
```

## 从模糊需求到可复查结论

```text
模糊问题：“看看导出有没有问题”
              ↓
可验证断言：“未登录用户点击导出会看到登录提示，且不会下载”
              ↓
确认环境：URL、浏览器、身份、测试数据、是否允许写入
              ↓
先观察 → 最小操作 → 保存前后证据 → 重载复查
              ↓
通过 / 失败 / 无法确认 + 复现步骤 + 限制
```

好的断言有四个部分：

| 要素 | 例子 |
| --- | --- |
| 前提 | 未登录用户、指定测试身份、特定浏览器 |
| 操作 | 点击导出、打开某页面、输入测试数据 |
| 结果 | 出现登录提示、跳转到某 URL、显示错误文案 |
| 边界 | 不提交、不付款、不发布、不修改真实数据 |

## 一份合格的证据记录，至少包括

```text
断言：
环境：URL、浏览器、登录身份、测试数据
时间：

步骤：
1.
2.

结果：通过 / 失败 / 无法确认
证据：截图或快照路径、关键 URL、可见提示
限制：缓存、权限、网络、第三方服务或未获授权的写入动作
```

日志只记录完成判断所需的信息。请不要写入真实密码、会话 cookie、令牌、订单信息或客户隐私数据。

## 重要边界：哪些操作必须先授权

以下行为在没有用户明确许可时都应停止在最终动作之前：

- 提交表单、发消息、触发邮件或通知
- 发布内容、创建订单、付款
- 删除数据、修改生产数据或覆盖设置
- 访问非用户授权的账号或绕过权限控制

遇到这类动作，正确做法是先报告“验证已到达最终操作前，执行此动作会产生外部影响，需要授权”，而不是自行继续。

同样，工具访问失败、页面要求登录、动态内容没加载，都只能说明当前无法确认；它们不等于产品故障。

## 常见问题

**它会替我操作网页吗？**  
会在你明确的验证范围内通过浏览器完成最小必要步骤。任何会影响外部系统的最终操作，必须先得到你明确授权。

**截图有了就算验证完成吗？**  
不一定。截图只是某一刻的可见状态；重要流程还应记录前置条件、操作、URL 和刷新/重开后的结果。

**没复现问题，能说问题不存在吗？**  
不能。只能说“当前条件下未复现”，并列出浏览器、账号、网络、缓存或数据上的差异。

**能拿去做持续回归测试吗？**  
它能帮助你定义断言和证据标准。持续自动化应使用项目已有的测试框架与配置，并避免将敏感凭据写入业务代码或日志。
