# 明辰浏览器验真

目标不是“浏览过页面”，而是交付别人可以复查的证据。任何结论都应关联明确的浏览器状态、操作步骤和可见结果。

## 验真流程

1. **定义断言**：把“看看是否正常”改写成可验证句子，例如“未登录用户点击导出会看到登录提示”。
2. **确认环境**：记录测试地址、浏览器、登录身份、必要的测试数据和是否允许写入外部系统。
3. **先观察再操作**：打开页面、保存初始快照，确认元素、错误状态和当前登录态；不要猜测选择器或跳过前置条件。
4. **执行最小路径**：每次只完成验证所需的操作。填写数据时使用测试数据；如果操作会提交、发布、付款、删除或触发通知，先获得用户明确许可。
5. **固定证据**：在关键前后保存截图或页面快照，并记录 URL、时间、操作、可见结果和失败信息。
6. **复查结论**：重载页面或重新打开会话，确认结果不是临时视觉状态；无法复现时如实报告条件和不确定性。

## 选择验证范围

| 用户目标 | 默认验证范围 |
| --- | --- |
| “页面能不能打开” | URL、首屏状态、可见错误、关键元素和截图 |
| “这个按钮有没有反应” | 点击前后状态、页面跳转或提示、重载后的结果 |
| “登录后能否使用某功能” | 登录身份、前置条件、最小操作路径和权限提示 |
| “这个问题能否复现” | 复现条件、步骤、预期与实际结果、稳定性 |
| “这个流程能不能上线” | 只验证用户明确列出的断言；发布、支付、删除等写操作必须另行授权 |

## 安全与保真

- 截图或日志中出现姓名、邮箱、订单、会话信息或其他敏感内容时，先遮盖或只记录必要摘要。
- 未获得明确许可时，停在会产生外部写入之前，例如提交表单、发消息、下单、发布、删除或修改生产数据。
- 不把“页面看起来像成功”写成“业务已完成”；需用结果页、状态字段、刷新后的状态或用户允许的后台证据确认。
- 无法验证时写清“无法确认”、所缺条件和已观察到的事实；不要将工具报错或访问限制解释成产品故障。

## 交付格式

按以下顺序输出：

1. **结论**：每条断言标注“通过、失败或无法确认”。
2. **复现步骤**：仅列出完成验证所需的最小路径。
3. **证据**：截图/快照位置、关键 URL、时间和可见结果。
4. **限制与风险**：说明权限、测试数据、网络、缓存、第三方服务或未授权写入造成的边界。

## 工具选择

- 需要快速检查、截图或提取可见文本时，使用当前环境提供的浏览器控制工具。
- 需要可持续维护的自动化回归测试时，使用项目已有的测试框架和配置。
- 需要临时脚本时，使用项目已有依赖；不要为了单次验证向业务代码写入凭据或大段测试基础设施。
